## 1. 版本定位

v0.2 是 PrismBlade 从“监看交互原型”转向“真实像素监看原型”的阶段。v0.1.x 的主要成果是横屏优先 UI、Mock 相机控制、LUT 导入、曝光模式锁定规则、底部控制条和基础交互闭环；v0.2 则把主线推进到 `CVPixelBuffer -> MTLTexture -> MTKView` 的 Metal-first 图像处理管线。

这一阶段的重点不是连接真实 Nikon Z6III，也不是实现 USB/PTP 通信，而是先把未来真实 live view 必须经过的图像链路搭起来：媒体帧、GPU texture、LUT、伪色、斑马纹、scope、视频文件验证源、状态持久化和自动化测试。

v0.2 可以视为三个连续切片：

- v0.2.1：建立 Metal-first 真实像素管线，替换 SwiftUI 合成预览和占位曝光辅助。
- v0.2.2：修正 N-Log / LUT / 曝光分析信号边界，避免 N-Log 技术 LUT 吃到已经 decode 过的错误输入。
- v0.2.3：改善 scope 可读性，包括更高 bins 分辨率、非线性密度归一化、白色 waveform、分析源标题和可拖动停靠。

## 2. 已完成的核心变化

### 2.1 媒体帧与输入边界

当前 `VideoFrame` 已经从 v0.1 的模拟动画状态升级为媒体帧模型，携带 `sequence`、`CMTime` timestamp、`FrameFormat`、`CVPixelBuffer` 和相机 metadata。`FrameSource` 只负责提供媒体帧，不关心 UI、相机传输或 Metal 实现。

代码中已存在两类帧源：

- `SimulatedFrameSource`：默认帧源，生成真实 BGRA `CVPixelBuffer`，用于模拟器持续预览。
- `VideoFileFrameSource`：使用 `AVAssetReader` 读取本地视频，输出同一套 `VideoFrame`，用于真实素材验证。

`AppEnvironment` 目前通过 `-PBLocalVideoPath` launch argument 临时注入 `VideoFileFrameSource`。这符合 v0.2 的验证需求，但还不是用户可见的视频文件播放入口。

### 2.2 Metal 预览主路径

主预览已由 SwiftUI gradient / overlay 占位切换为 `MTKView`：

```text
VideoFrame.pixelBuffer
  -> MetalTextureBridge
  -> source MTLTexture
  -> PreviewShaders.metal
  -> MTKView drawable
```

相关实现包括：

- `MetalTextureBridge`：使用 `CVMetalTextureCache` 将 BGRA pixel buffer 桥接为 `.bgra8Unorm` texture。
- `MetalPreviewRenderer`：作为 `MTKViewDelegate` 管理 command queue、render pipeline、viewport、LUT resource 和 scope compute 编排。
- `MetalPreviewSurface`：通过 `UIViewRepresentable` 把 `MTKView` 接回 SwiftUI。
- `MetalFrameProcessor`：集中解析 LUT 状态、生成 uniforms，并触发 scope compute 旁路。

这条路径已经建立了后续真实 camera live view 可以替换接入的关键边界。

### 2.3 LUT 链路

v0.2 已经完成从 `.cube` 文本解析到 Metal 3D texture 采样的闭环：

- `LUTParser` 校验 `.cube` 文件结构和数据行。
- `LUTRepository` 负责导入 LUT、加载本地可选 LUT 和持久化 descriptor。
- `LUTStore` 作为运行时 LUT 读取和缓存入口。
- `LUTPass` 将解析后的 entries 上传为 `.rgba32Float` 3D texture。
- `PreviewShaders.metal` 在 fragment shader 中采样 LUT，并按 intensity 混合 raw input 与 LUT output。

v0.2.2 后，N-Log LUT 预览会直接采样原始 N-Log 码值，不再走“先 `nlogDecode()` 再进 N-Log -> Rec.709 LUT”的错误路径。当前代码中 `MetalFrameProcessor.resolveLUTRenderState` 只在 `frame.format.colorEncoding == .nLog` 且 LUT 开启、强度大于 0、存在可解析选中 LUT 时启用 LUT；Rec.709 / HLG 的 display-space 或 creative LUT 策略尚未展开。

### 2.4 曝光辅助与分析源

v0.2.2 把 shader 内部信号拆成三层：

```text
rawInputColor
previewColor
analysisColor
```

`ExposureAnalysisSource` 目前支持：

- `Raw Signal`：默认值，scope / zebra / false color 分析原始输入信号。
- `Preview Display`：分析 LUT 后或当前预览显示信号。

这解决了 N-Log 记录曝光与 LUT 后监看观感混在一起的问题。用户可以在 Settings 中切换分析源，scope 标题和 N-Log 状态栏也会显示当前分析语义。

伪色和斑马纹已经从 SwiftUI 占位 overlay 升级为 shader 级处理。它们基于所选 `analysisColor` 计算 luma，再映射 false color 或 zebra mask。当前阈值仍是 normalized signal / Rec.709 风格的简化解释，尚未针对 Nikon N-Log 灰卡、肤色或高光保护线做最终校准。

### 2.5 Scope compute 与绘制

Luma waveform 和 RGB Parade 已经从程序曲线升级为 Metal compute bins：

```text
source texture
  -> raw / preview / analysis signal selection
  -> ScopeComputePass
  -> compact ScopeData bins
  -> ScopePanel Canvas drawing
```

v0.2.3 完成的可读性更新包括：

- 默认 bins 从 `96 x 64` 提升到 `192 x 96`。
- 最大采样提升到 `640 x 360`。
- 默认 `frameInterval = 3`，目标约 10fps scope 分析。
- readback 使用 `log1p(count) / log1p(maximum)`，避免低密度信号被高密度区域压没。
- Luma waveform 改为白色，RGB Parade 保持 RGB 分色。
- 标题显示 `Waveform · Raw`、`RGB Parade · LUT` 等分析源信息。
- scope 面板支持四角拖动吸附，并通过 `UserDefaults` 持久化停靠位。
- 参数调整浮层打开时，scope 会自动上移避让底部控制区域。

### 2.6 UI、设置与 Mock 相机

v0.1 已有的监看 UI 在 v0.2 中继续保留，并接入了真实 Metal 管线状态：

- 顶部状态栏显示连接状态、分辨率、帧率、输入编码、N-Log Raw/LUT 状态、曝光辅助、电量和存储占位。
- 左右工具栏控制伪色、斑马、scope、N-Log LUT Preview、LUT 管理、缩放和设置。
- 底部相机控制条继续走 `CameraCommandService -> CameraTransport` 边界。
- Mock 相机仍支持曝光模式、光圈、快门、ISO、白平衡、对焦模式、录制、拍照和对焦。
- 曝光模式参数锁定规则在 UI、service 和 transport 三层校验。
- Settings 已包含竖屏监看、伪色/斑马默认值、斑马阈值、scope 透明度、scope 模式、scope 停靠位置、曝光分析源和 Mock 调试入口。

## 3. 自动化测试现状

当前仓库已经建立 `PrismBladeTests` XCTest target，测试覆盖从 v0.1 的领域规则扩展到了 v0.2 的图像处理边界。按测试文件统计，目前有 60 个测试用例，覆盖：

- Pixel buffer fixture 生成。
- `VideoFrame` 媒体帧模型。
- `SimulatedFrameSource` 真实 pixel buffer 输出。
- `VideoFileFrameSource` 临时 `.mov` 读取、timestamp 和色彩编码识别。
- `MetalTextureBridge` BGRA pixel buffer 桥接。
- `.cube` parser 正常与异常路径。
- `LUTRepository` 本地 LUT 发现和导入 LUT reload。
- `LUTPass` 3D texture 上传顺序、domain 保留和 texture format。
- CPU reference：Rec.709 luma、LUT 采样、waveform / parade bins、N-Log / HLG decode reference。
- Metal preview shader：LUT intensity、N-Log raw 输出、raw N-Log LUT 采样、raw/LUT 混合、伪色、raw/preview zebra 和 false color 分析源。
- `ScopeComputePass`：默认配置、非线性归一化、waveform / RGB Parade bins、scope off、readback 节流、raw N-Log 和 Preview Display 分析源。
- 相机曝光模式规则、command service 和 Mock transport 非法写入拒绝。

README 中记录过 App 构建、测试构建和聚焦 `ScopeComputePassTests` 的成功验证。完整 XCTest 仍依赖可用 iOS Simulator、CoreSimulator / `testmanagerd` 和 Metal Toolchain。

## 4. 仍需后续更新的内容

### 4.1 真实素材校准

v0.2 已经有算法和渲染链路，但 Nikon Z6III 的真实监看可信度还需要真实素材校准：

- 使用 `material/NLOG.MOV` 和本地 Nikon N-Log -> Rec.709 LUT 验证 LUT 关闭时为平的 N-Log，LUT 开启时不过早剪高光。
- 使用 Rec.709、N-Log、HLG 的灰卡、色卡、肤色、欠曝、过曝、暗部噪声素材做对比。
- 与 DaVinci Resolve、Final Cut、Nikon 官方查看器或外部监视器的 LUT 后画面和 waveform / parade 截图对照。
- 为 Raw N-Log waveform 增加中灰、肤色和高光保护参考线。
- 明确 HLG 的 SDR/HDR 监看策略，目前 HLG 仍采用较保守的 raw-signal-first 行为。

### 4.2 LUT metadata 与兼容策略

当前 LUT 应用策略偏向 Nikon N-Log 技术 LUT。后续需要补齐：

- `LUTInputSpace` / `LUTOutputSpace` metadata。
- Technical LUT 与 Creative LUT 的区分。
- 按 `FrameFormat.colorEncoding` 自动建议可用 LUT。
- 对 Rec.709 输入启用 N-Log LUT 时给出风险提示或禁用。
- 识别 LUT domain、cube size、目标显示空间和厂商来源。
- 支持 LUT 缩略图、分组、收藏或搜索。

### 4.3 Pixel format 与视频输入

`MetalTextureBridge` 当前只支持 `kCVPixelFormatType_32BGRA`，`VideoFileFrameSource` 也强制 AVAssetReader 输出 BGRA。这让 v0.2 管线稳定，但后续真实 live view 或更高性能路径可能需要：

- 支持 YCbCr / bi-planar pixel buffer。
- 减少解码到 BGRA 的额外转换成本。
- 明确 full range / video range 处理。
- 处理 HDR metadata、color primaries、transfer function 和 matrix。
- 为真实相机 live view 引入专用 `FrameSource`，并决定是否继续使用 `VideoFrame` 作为唯一媒体边界。

### 4.4 Scope 专业化

v0.2.3 解决了“可见、可读”的问题，但还不是完整专业监视器 scope：

- 增加 waveform 标尺。
- 增加 N-Log / Rec.709 / HLG 不同分析源下的参考线。
- 提供 scope 质量 / 性能档位，例如 `192 x 96` 与 `256 x 128`。
- 在 iPhone 12 Pro 真机上验证 `frameInterval`、采样上限和 readback 开销。
- 增加 histogram。
- 支持 LUT 前 / LUT 后 scope 并排或更明确的模式切换。
- 继续改善低密度 waveform 的线条感和避免过度 block 化。

### 4.5 曝光辅助专业化

当前 false color 和 zebra 已经走 shader，但映射仍是 v0.2 原型级：

- 为 N-Log raw signal 定义独立阈值说明，不再混用 Rec.709 IRE 直觉。
- 支持 false color 映射表配置或监视器风格 presets。
- 为 zebra 增加更明确的 high / range UI 文案和 preview display/raw signal 解释。
- 用真实肤色、灰卡和高光素材校准默认阈值。
- 增加对 clipping、super white、video range 的处理策略。

### 4.6 真实相机通信

v0.2 明确没有实现真实 Nikon 连接。下一阶段需要从当前边界继续推进：

- 增加 ImageCaptureCore、PTP、Network Bridge 的 adapter 命名空间或空实现。
- 验证 iPhone 12 Pro 与 Nikon Z6III 的 USB 模式，包括 Nikon `iPhone` 模式、MTP/PTP、USB-LAN 或 Wi-Fi/桥接方案。
- 判断 ImageCaptureCore 是否满足发现、媒体访问、拍摄触发和参数控制。
- 评估 PTP / `libgphoto2` 思路在 iOS 上的权限、构建和 App Store 合规风险。
- 保持 SwiftUI 和 Metal 管线不直接依赖 Nikon、USB、PTP 或 `libgphoto2` 类型。
- 为真实相机能力表、曝光模式读取、参数可写性和错误恢复建立模型。

### 4.7 用户可见工作流

v0.2 的真实视频源主要是开发验证入口，后续产品体验还需要：

- 决定是否提供用户可见的视频文件测试入口，或只保留为开发模式。
- 增加真实素材选择、最近文件或测试素材面板。
- 增加错误状态页或预览失败 overlay，避免 Metal bridge 失败时只清屏。
- 增加性能指标展示，例如渲染帧率、scope 分析间隔、readback 耗时。
- 增加对焦峰值。
- 增加真实录制、拍照文件保存或导入流程。

## 5. 建议的后续开发顺序

1. 先冻结 v0.2 的核心边界：`FrameSource`、`VideoFrame`、`MetalPreviewRenderer`、`MetalFrameProcessor`、`CameraTransport` 暂时不做大重构，只修必要 bug。
2. 用真实 Rec.709 / N-Log / HLG 素材完成手动验证，优先确认 N-Log LUT 输入、Raw Signal 默认分析和 scope 可读性。
3. 增加 LUT metadata，解决 technical LUT / creative LUT / 输入空间兼容性问题。
4. 做 N-Log / HLG / false color / zebra / waveform 的真实素材校准。
5. 增加 scope 标尺、参考线和性能档位。
6. 建立真实相机 adapter 空壳，再开始硬件通信预研。
7. 在 iPhone 12 Pro 真机上验证 Metal 性能、scope readback、方向切换和 UI 遮挡。

## 6. 阶段结论

v0.2 的功能开发可以阶段性收束。当前仓库已经不再只是“看起来像监看”的 SwiftUI 原型，而是具备真实 pixel buffer 输入、Metal 预览、3D LUT、raw/preview 分析源、shader 曝光辅助和 compute scope 的图像处理原型。

后续最大的风险不在 UI 搭建，而在真实相机和真实素材带来的校准与兼容性：Nikon Z6III 的 live view 获取路径、N-Log / HLG 的参考解释、LUT input/output metadata、pixel format / range 处理、真机性能和硬件连接能力。只要继续保持 v0.2 已经建立的边界清晰，后续可以把真实相机通信和颜色校准逐步替换进来，而不需要推倒当前监看界面和 Metal 管线。
